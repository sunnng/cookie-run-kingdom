# 项目任务调度说明文档

## 1. 概述

本项目采用「单线程串行调度 + 弹窗守卫」的架构：

- **串行调度**：同一时刻只执行一个业务任务，避免多任务并发导致的状态混乱。
- **条件触发**：每个任务注册时提供 `condition` 函数，调度器每轮按顺序检查，第一个满足条件的任务执行。
- **弹窗守卫**：主循环和每次长等待期间都会扫描屏幕弹窗，命中后自动点击处理。
- **空闲等待**：没有任务可执行时，根据最近的计时器（矿山远距/开采/海滩补货）进入等待，而非空转。

## 4. 运行时主循环

`Runtime.run()` 启动后进入一个 `while true` 死循环：

```lua
while true do
    round = round + 1
    Guard.check()                    -- 1. 先清弹窗
    local hasWork, ok = Scheduler.run(rt.STOP_ON_ERROR)  -- 2. 执行一轮调度

    if not ok then return end        -- STOP_ON_ERROR 时终止

    if not hasWork then
        -- 无任务：进入空闲等待
        idleWaitLogic()
    else
        -- 有任务：轮间小睡
        Guard.sleep(stepMs, guardStep)
    end
end
```

### 4.1 时间参数

参数来自 `config.lua` 的 `RUNTIME`：

| 参数 | 默认值 | 说明 |
|---|---|---|
| `GUARD_INTERVAL_MS` | 500ms | 长等待分片间隔，每片调用一次 `Guard.check` |
| `STEP_DELAY_MS` | 5000ms | 本轮有任务执行后，下轮前的等待时间 |
| `IDLE_DELAY_MS` | 30000ms | 无任务时基础挂机等待时间 |
| `STOP_ON_ERROR` | false | 任务 action 异常时是否终止整个脚本 |

## 5. 调度器

### 5.1 API

```lua
Scheduler.add(name, condition, action)
Scheduler.clear()
Scheduler.count()
Scheduler.run(stopOnError)
```

### 5.2 单轮执行规则

1. 按 **注册顺序** 遍历任务。
2. 对每个任务执行 `pcall(condition)`：
   - 异常 → 记录 warn，继续下一个。
   - 返回 `false` → 跳过，记录到 skipped 列表。
   - 返回 `true` → 执行该任务的 `action`。
3. 一旦某个任务被执行，**不会继续检查后面任务**（串行）。
4. `action` 执行后记录耗时和结果。
5. 返回 `hasWork`（本轮是否有任务执行）。

### 5.3 执行顺序（注册顺序 = 优先级）

`game/register.lua` 中注册顺序：

1. 矿山勘查
2. 矿山开采
3. 海滩交易所
4. 王国竞技场
5. 布谷鸟广场
6. 洗脆饼词条

因此：

- **矿山任务优先级最高**，只要条件满足就会先执行。
- **海滩交易所、王国竞技场、布谷鸟广场** 在执行前会调用 `isMineSchedulerIdle()`，若矿山任务未到期则主动跳过，让出 CPU 给矿山。
- **洗脆饼词条** 不参与矿山让渡，只要开启就会执行。

### 5.4 任务条件设计

每个 `condition` 函数的典型结构：

```lua
function()
    -- 1. 功能开关
    if not UserConfig.get("xxx").enabled then
        return false
    end

    -- 2. 高优先级任务是否空闲（可选）
    if not isMineSchedulerIdle() then
        return false
    end

    -- 3. 自身业务状态是否就绪
    if not SomeSession.checkReady() then
        return false
    end

    return true
end
```

### 5.5 Action 执行约定

- `action` 内部自行管理状态机或流程。
- 返回 `false` 会被记录为 warn（通常表示任务主动失败）。
- 异常会被 `pcall` 捕获，根据 `STOP_ON_ERROR` 决定是否终止脚本。

## 6. 矿山优先策略

### 6.1 `isMineSchedulerIdle()`

```lua
local function isMineSchedulerIdle()
    if 矿山勘查开启 and Session.checkFarWait() then
        return false
    end
    if 矿山开采开启 and MiningSession.checkReady() then
        return false
    end
    return true
end
```

含义：只要矿山勘查或矿山开采有一个「可以执行」，就认为矿山侧不空闲，其他非洗脆饼任务要让路。

### 6.2 让渡任务清单

| 任务 | 是否调用 isMineSchedulerIdle |
|---|---|
| 矿山勘查 | 否（自身就是最高优先级） |
| 矿山开采 | 否（自身就是最高优先级） |
| 海滩交易所 | 是 |
| 王国竞技场 | 是 |
| 布谷鸟广场 | 是 |
| 洗脆饼词条 | 否 |

## 7. 弹窗守卫（Guard）

### 7.1 机制

- `Guard.register(name, feature, handler, priority)` 注册一个陷阱。
- `Guard.check()` 按 priority 降序扫描所有陷阱。
- 命中第一个匹配的陷阱后，执行其 `handler`，然后返回。
- `Guard.sleep(ms, stepMs)` 将长等待切成小片，每片调用 `Guard.check`。

### 7.2 当前注册陷阱

`game/register.lua` 中只注册了一个：

| 名称 | 特征 | 处理 | 优先级 |
|---|---|---|---|
| 网络联机状态不稳定 | `PopupPage["网络联机状态不稳定"]["特征"]` | 点击确认按钮 | 10 |

### 7.3 Guard Hook

`Runtime.run()` 中调用：

```lua
Color.setGuardHook(Guard.check)
```

这样 `lib/color.lua` 内部所有 `wait`/`waitMatch`/`tapUntilMatch`/`sleepFragment` 都会自动触发守卫扫描，确保长等待期间也能处理弹窗。

## 8. 空闲等待逻辑

当 `Scheduler.run` 返回 `hasWork = false` 时，进入空闲逻辑：

```lua
local farRemain   = 矿山勘查远距等待剩余秒
local miningRemain = 矿山开采 busy 等待剩余秒
local marketRemain = 海滩交易所补货等待剩余秒
local arenaRemain  = 王国竞技场免费刷新等待剩余秒
local waitRemain   = max(farRemain, miningRemain, marketRemain, arenaRemain)

if waitRemain > 0 then
    -- 有计时器未到期：每秒检查，刷新 HUD
    for _ = 1, idleSec do
        if 到期 then break end
        StatusHud.setMineWait({surveySec, miningSec, marketSec})
        Guard.sleep(1000, guardStep)
    end
else
    -- 无任何计时器：直接睡 30s
    StatusHud.setIdle()
    Guard.sleep(idleMs, guardStep)
end
```

### 8.1 设计目的

- 避免空转占用 CPU。
- 矿山/海滩/竞技场倒计时到期时能立即唤醒执行。
- HUD 实时显示剩余等待时间。

## 9. 状态栏（StatusHud）

调度过程中 HUD 的显示来源：

| 场景 | 调用 | 显示内容 |
|---|---|---|
| 启动 | `StatusHud.init` | `帅斌饼干 · 就绪` |
| 运行时 | `StatusHud.set("run", "运行中")` | `帅斌饼干 · 运行中` |

| 纯空闲 | `setIdle` | `帅斌饼干 · 空闲 · 等待任务` |

## 10. 扩展指南

### 10.1 新增一个调度任务

步骤：

1. 创建业务模块，暴露 `Task.run()` 或类似的 action 函数。
2. 在 `game/register.lua` 中 `require` 该模块。
3. 调用 `Scheduler.add("任务名", condition, action)` 注册。
4. 决定优先级位置：
   - 需要最高优先级 → 放在矿山任务附近。
   - 需要让渡矿山 → 在 condition 中调用 `isMineSchedulerIdle()`。
   - 后台/低优先级 → 放在布谷鸟广场之后。

示例：

```lua
local MyTask = require("game.常规_我的任务.任务")

Scheduler.add("我的任务",
    function()
        if not UserConfig.get("myTask").enabled then
            return false
        end
        if not isMineSchedulerIdle() then
            return false
        end
        return MyTask.canRun()
    end,
    function()
        if SquareRoute.isSquareContext() then
            SquareTask.leaveForOtherTask()
        end
        MyTask.run()
    end)
```

### 10.2 新增一个守卫陷阱

```lua
Guard.register("弹窗名称", feature, function()
    -- 处理弹窗，如点击确认
    tap(x, y)
end, priority)
```

注意：

- `feature` 可以是 `core.guard` 支持的比色特征表，也可以是返回 boolean 的函数。
- `priority` 越高越先检查。
- handler 内部不要抛异常，否则该次守卫处理失败。

## 11. 常见问题与调试

### 11.1 为什么我的任务没执行？

按顺序排查：

1. 功能开关是否开启（`UserConfig.get("xxx").enabled`）。
2. 是否有更高优先级任务先命中（看日志 `[执行] xxx 开始`）。
3. 是否被 `isMineSchedulerIdle()` 跳过（看日志 `矿山待执行，跳过 xxx`）。
4. 自身 `condition` 中的业务状态是否就绪（会话、倒计时等）。
5. `Scheduler.run` 是否只执行了前面某个任务就结束（串行特性）。

### 11.2 日志关键字

| 关键字 | 含义 |
|---|---|
| `[Scheduler] [执行] xxx 开始` | 任务开始执行 |
| `[Scheduler] [轮次] 无任务执行` | 本轮没有任务命中 |
| `[Register] 矿山待执行，跳过 xxx` | 任务因矿山未到期而主动跳过 |
| `[Runtime] [idle] 等待 剩余Xs` | 空闲等待中 |
| `[Guard] [命中] xxx` | 弹窗被命中 |
| `[Guard] [处理] xxx 完成` | 弹窗已处理 |

### 11.3 调整调度间隔

修改 `config.lua`：

```lua
RUNTIME = {
    STEP_DELAY_MS = 5000,   -- 有任务后轮间等待
    IDLE_DELAY_MS = 30000,  -- 无任务时挂机等待
    GUARD_INTERVAL_MS = 500, -- 守卫分片间隔
}
```
