# 弹窗模块使用说明文档

## 一、概述

弹窗模块负责识别并自动处理游戏运行过程中出现的各类弹窗，包括：

- **业务流程弹窗**：玩法内部确定会出现的确认/取消弹窗
- **偶发守卫弹窗**：网络异常、登录提示、维护公告等不可预期的通用弹窗
- **链式/分支弹窗**：一次操作后可能连续触发多个弹窗，或主弹窗后分支出次弹窗

模块由两层组成：

| 模块 | 路径 | 职责 |
|------|------|------|
| `Dialog` | `lib.dialog` | 通用弹窗对象：识别、点击、等待消失、复杂消解 |
| `Guard` | `core.guard` | 全局陷阱守卫：周期性扫描并自动处理高优先级通用弹窗 |

## 二、快速开始

### 2.1 定义一个弹窗

弹窗由以下要素构成：

```lua
local Dialog = require("lib.dialog")

local confirmDialog = Dialog.new({
    name = "购买确认",
    feature = { "x|y|color,...", 0.9 },      -- 比色特征
    confirmBtn = { 800, 600, 900, 650 },     -- 确认按钮区域
    cancelBtn = { 700, 600, 780, 650 },      -- 取消按钮区域（可选）
    neverAgainBtn = { 500, 500, 600, 530 },  -- 今日不再显示（可选）
}, { tag = "[海滩交易所]" })
```

字段兼容多种命名：

| 字段 | 兼容写法 |
|------|----------|
| `name` | `name` / `名称` |
| `feature` | `feature` / `特征` |
| `confirmBtn` | `confirmBtn` / `confirm` / `buyBtn` / `按钮_确认` |
| `cancelBtn` | `cancelBtn` / `cancel` / `closeBtn` |
| `neverAgainBtn` | `neverAgainBtn` / `todayNotAskAgain` |

### 2.2 判断弹窗是否可见

```lua
if confirmDialog:isVisible() then
    -- ...
end
```

### 2.3 等待弹窗出现或消失

```lua
confirmDialog:waitAppear(5000, 300)  -- 5秒内每隔300ms检测
confirmDialog:waitGone(3000, 300)    -- 等待弹窗消失
```

### 2.4 直接点击按钮

```lua
confirmDialog:tapConfirm(800)    -- 点击确认按钮，等待800ms
confirmDialog:tapCancel(800)     -- 点击取消按钮
confirmDialog:tapNeverAgain(800) -- 勾选"今日不再显示"选择框（不会点击确认/取消）
confirmDialog:tap("confirm", 800) -- 通用点击接口
```

> 注意：`tapNeverAgain` 只负责勾选"今日不再显示"复选框，勾选后仍需调用 `tapConfirm` 或 `tapCancel` 关闭弹窗。

## 三、Dialog:handle 单弹窗处理

`handle` 是最常用的弹窗处理接口，支持两种模式。

### 3.1 flow 模式（业务流程弹窗）

用于玩法流程中**应当出现**的弹窗。可配置等待弹窗出现，未出现时根据 `required` 决定是否报错。

```lua
local ok, reason = confirmDialog:handle({
    mode = "flow",
    action = "confirm",       -- 主动作：confirm 或 cancel
    neverAgain = true,        -- 可选：先勾选"今日不再显示"，再执行 action
    waitAppearMs = 5000,      -- 等待弹窗出现的最长时间
    waitGoneMs = 3000,        -- 点击后等待消失的最长时间
    required = true,          -- true: 未出现则返回失败; false: 未出现则跳过
    tapDelayMs = 800,         -- 点击后等待时间
    intervalMs = 500,         -- 轮询间隔
})
```

`action` 只能是 `confirm` 或 `cancel`；`neverAgain` 是附加勾选动作，不是关闭弹窗的动作。

### 3.2 ifVisible 模式（Guard / 偶发弹窗）

用于弹窗**可能出现也可能不出现**的场景。如果当前不可见，直接返回成功。

```lua
local ok, reason = confirmDialog:handle({
    mode = "ifVisible",
    action = "cancel",
    waitGoneMs = 2000,
})
```

### 3.3 转换为 Guard handler

```lua
local Guard = require("core.guard")

Guard.register(
    "网络联机状态不稳定",
    unstableDialog.def.feature,
    unstableDialog:toGuardHandler({ action = "confirm", waitGoneMs = 2000 }),
    10
)
```

## 四、Guard 全局守卫

### 4.1 注册守卫

```lua
local Guard = require("core.guard")
local Dialog = require("lib.dialog")

local d = Dialog.new({
    name = "网络联机状态不稳定",
    feature = { "...", 0.9 },
    confirmBtn = { 775, 621, 828, 647 },
}, { tag = "[Register]" })

Guard.register("网络联机状态不稳定", d.def.feature, d:toGuardHandler({
    action = "confirm",
    waitGoneMs = 2000,
}), 10)
```

`priority` 越大优先级越高，同优先级按名称字典序稳定排序。

### 4.2 手动触发扫描

```lua
Guard.check()          -- 扫描并按优先级处理首个命中弹窗
Guard.sleep(3000, 500) -- 分片 sleep，每 500ms 扫描一次
```

### 4.3 清空守卫

```lua
Guard.clear()
```

通常在 `Runtime` 启动时先 `Guard.clear()` 再重新注册。

## 五、复杂弹窗场景

### 5.1 顺序未知的 0~N 个弹窗：resolveUntilIdle

适用于一次操作后可能连续弹出多个弹窗，但顺序不确定。

```lua
local Dialog = require("lib.dialog")

local deployDialog = Dialog.new(F.dialog.deployMore, { tag = TAG })
local toppingDialog = Dialog.new(F.dialog.missingTopping, { tag = TAG })

local ok, summary = Dialog.resolveUntilIdle({
    { dialog = deployDialog, name = "deployMore", priority = 10,
      opts = { action = "confirm", waitGoneMs = 1000, intervalMs = 500 } },
    { dialog = toppingDialog, name = "missingTopping", priority = 10,
      opts = { action = "confirm", waitGoneMs = 1000, intervalMs = 500 } },
}, {
    timeoutMs = 8000,    -- 总超时
    minWaitMs = 500,     -- 至少等待多久
    settleMs = 800,      -- 连续多久无弹窗视为稳定
    maxHandled = 2,      -- 最多处理几个弹窗（可选，≤0 表示无限）
    tag = TAG,
})

if not ok then
    Logger.warn(TAG .. " 弹窗处理失败 | " .. tostring(summary.reason))
end
```

说明：

- `resolveUntilIdle` 内部使用原生 `sleep` 轮询，**不再触发 Guard.check**，保证弹窗消解过程的原子性。
- 处理完成后返回 `summary`，包含 `handled`（处理数量）、`names`（处理名称列表）、`lastReason`（最后 reason）。
- `maxHandled` 设置为 `nil` 或 `≤0` 表示不限制数量。

### 5.2 主弹窗后分支出次弹窗：resolveAfterPrimary

适用于先处理一个必现主弹窗，然后可能出现次弹窗或直接进入成功状态。

```lua
local confirmDialog = Dialog.new(DialogDef, { tag = TAG })
local shortageDialog = Dialog.new(ShortageDialogDef, { tag = TAG })

local ok, outcome, reason = Dialog.resolveAfterPrimary({
    primary = {
        dialog = confirmDialog,
        opts = {
            mode = "flow",
            action = "confirm",
            waitAppearMs = 5000,
            required = true,
        },
    },
    watch = {
        {
            dialog = shortageDialog,
            opts = { action = "cancel", waitGoneMs = 2000 },
            result = "shortage",
            after = function()
                -- 次弹窗处理后的额外逻辑
            end,
        },
    },
    successWhen = function()
        return not confirmDialog:isVisible()
    end,
    successResult = "purchased",
    timeoutMs = 5000,
    tag = TAG,
})

if outcome == "purchased" then
    -- 购买成功
elseif outcome == "shortage" then
    -- 道具不足
end
```

### 5.3 顺序确定的弹窗链：handleChain

适用于弹窗按固定顺序出现。

```lua
Dialog.handleChain({
    { def = dialogA, opts = { action = "confirm", waitGoneMs = 1000 } },
    { def = dialogB, opts = { action = "cancel", waitGoneMs = 1000 } },
}, { tag = TAG })
```

## 六、通用弹窗管理

项目中将游戏中跨场景通用的弹窗集中定义在 `game/通用_弹窗/page.lua`，并在 `game/register.lua` 注册为 Guard。

### 6.1 新增通用弹窗步骤

1. 在 `game/通用_弹窗/page.lua` 中定义弹窗特征和按钮：

```lua
local 维护公告 = {
    特征 = { "...", 0.9 },
    按钮_确认 = { x1, y1, x2, y2 },
}
DialogPage.维护公告 = 维护公告
```

2. 在 `game/register.lua` 中注册：

```lua
local maintenancePopup = PopupPage["维护公告"]
local maintenanceDialog = Dialog.new({
    name = "维护公告",
    feature = maintenancePopup["特征"],
    confirmBtn = maintenancePopup["按钮_确认"],
}, { tag = "[Register]" })

Guard.register(
    "维护公告",
    maintenanceDialog.def.feature,
    maintenanceDialog:toGuardHandler({ action = "confirm", waitGoneMs = 2000 }),
    5
)
```

## 七、最佳实践

1. **通用偶发弹窗走 Guard，玩法内确定弹窗走 Dialog:handle / resolveUntilIdle / resolveAfterPrimary。**
2. **不要直接调原生 `sleep` 做长等待**，用 `Guard.sleep` 或 `Color.wait` 系列，保证等待期间能处理弹窗。
3. **比色特征集中到 `*_特征库.lua`**，不要在业务代码里散落比色字符串。
4. **弹窗按钮坐标/区域也集中到特征库**，便于维护和适配分辨率变更。
5. **`resolveUntilIdle` 已保证原子性**，其运行期间不触发 Guard，适合处理玩法内已知弹窗集合。
6. **Guard 优先级合理分配**：网络异常、连接中断等阻断性弹窗优先级应高于普通提示。
7. **处理失败时查看 `reason`**：`not_visible`（未出现）、`not_gone`（点击后未消失）、`no_confirm_btn`（按钮未配置）等。

## 八、常见错误

| 错误 | 原因 | 解决 |
|------|------|------|
| 弹窗未被识别 | `feature` 比色字符串错误 / 相似度不够 | 用 `Color.match` 单独验证 |
| 点击后弹窗不消失 | `waitGoneMs` 太短 / 按钮坐标偏移 | 增加 `waitGoneMs` 或校验按钮区域 |
| Guard 弹窗与业务弹窗冲突 | 业务弹窗未走 `lib.dialog`，各自点击不同按钮 | 统一用 `Dialog` 抽象 |
| `resolveUntilIdle` 超时 | 弹窗未在 `candidates` 中配置 / `settleMs` 太小 | 补全候选弹窗或调整参数 |
| `maxHandled = 0` 不处理弹窗 | 已修复：`maxHandled` 需 `> 0` 才生效 | 传 `nil` 或正整数 |
| `invalid_action` | `tap` 或 `handle` 传了 `action = "neverAgain"` | `neverAgain` 应作为独立选项 `neverAgain = true` 传入 |

## 九、API 速查

### Dialog 对象

| 方法 | 说明 |
|------|------|
| `Dialog.new(def, opts)` | 创建弹窗对象 |
| `d:isVisible()` | 当前是否可见 |
| `d:waitAppear(ms, interval)` | 等待出现 |
| `d:waitGone(ms, interval)` | 等待消失 |
| `d:tapConfirm(delayMs)` | 点击确认按钮 |
| `d:tapCancel(delayMs)` | 点击取消按钮 |
| `d:tapNeverAgain(delayMs)` | 勾选"今日不再显示"复选框 |
| `d:tap(action, delayMs)` | 通用点击，`action` 仅支持 `confirm` / `cancel` |
| `d:handle(opts)` | 单弹窗处理 |
| `d:toGuardHandler(opts)` | 生成 Guard handler |

### 静态方法

| 方法 | 说明 |
|------|------|
| `Dialog.handleChain(items, defaultOpts)` | 顺序确定的弹窗链 |
| `Dialog.resolveUntilIdle(candidates, opts)` | 顺序未知的 0~N 弹窗 |
| `Dialog.resolveAfterPrimary(cfg)` | 主弹窗后分支 |

### Guard

| 方法 | 说明 |
|------|------|
| `Guard.register(name, feature, handler, priority)` | 注册守卫 |
| `Guard.clear()` | 清空所有守卫 |
| `Guard.check()` | 手动扫描一次 |
| `Guard.sleep(ms, stepMs)` | 分片 sleep，期间扫描 |
| `Guard.trapCount()` | 当前守卫数量 |
