# 王国竞技场模块说明文档

## 1. 模块概述

王国竞技场模块用于自动化完成《冲呀！饼干人：王国》中的「王国竞技场」日常对战。模块支持自动进入竞技场、扫描对手、按奖杯差过滤、自动战斗、胜负统计、自动买票、免费刷新以及安全离开。

---

## 2. 目录结构

```
帅斌饼干/脚本/game/常规_王国竞技场/
├── 竞技场_特征库.lua   # UI 特征与坐标配置
├── 竞技场_页面.lua     # 页面识别、交互、战斗流程
├── 竞技场_路由.lua     # 导航：进入 / 离开竞技场
├── 竞技场_任务.lua     # 任务编排与扫荡策略
└── 竞技场_会话.lua     # 运行时状态管理
```

## 3. 模块职责

### 3.1 竞技场_特征库.lua

集中定义竞技场相关的所有图像特征、OCR 区域与点击坐标。当游戏 UI 改版或分辨率适配时，通常只需要修改此文件。

主要配置分区：

| 分区 | 说明 |
|---|---|
| `lobby` | 大厅特征、关闭按钮、奖牌/门票 OCR、奖杯 OCR、刷新按钮 OCR、买票按钮与滑条 |
| `opponent` | 对手查找特征、战力/奖杯 OCR 区域、已战结果颜色识别 |
| `teamSelect` | 队伍选择页特征、开始战斗按钮 |
| `dialog` | 「部署更多饼干」「未装载配料」弹窗特征与确认按钮 |
| `battle` | 战斗开始 UI 特征（当前仅作辅助） |
| `settlement` | 结算页特征、结果 OCR、离开按钮、升段跳过坐标 |
| `pagination` | 左右滑翻页坐标 |

### 3.2 竞技场_页面.lua

封装所有与竞技场 UI 的交互操作，向上层提供语义化 API。

核心 API：

| 函数 | 说明 |
|---|---|
| `isLobby()` / `waitLobby(timeout)` / `ensureLobby()` | 大厅检测与恢复 |
| `readMedalAndTicket()` | OCR 读取奖牌与门票数量 |
| `readTrophyCount()` | OCR 读取当前奖杯数 |
| `readOpponentInfo(location)` | 读取指定位置对手的战力、奖杯、是否已战、战斗结果 |
| `swipePageLeft()` / `swipePageRight()` | 对手列表翻页 |
| `runBattle()` | 完整单场战斗：进队伍选择 → 开始 → 等结算 → OCR 结果 → 离开结算（含升段跳过） |
| `isFreeRefresh()` / `waitFreeRefresh()` | 检测并点击免费刷新 |
| `buyTicket()` | 购买竞技场门票 |
| `ensureLobby()` | 确保当前在大厅；会依次尝试离开结算页、跳过升段奖励 |

### 3.2.1 离开结算页与升段奖励跳过

`runBattle()` 结算后调用 `leaveSettlement()` 完成「结算页 → 大厅」的完整流程，`ensureLobby()` 则作为兜底恢复手段。

`leaveSettlement()` 执行步骤（参考历史实现优化）：

1. **已在大厅**：直接返回。
2. **不在结算页**：返回 `false`，由调用方转交 `skipPromotion()` 处理。
3. **点击直到出现离开按钮**：使用 `retryTap(1460, 812, hasLeaveBtn, 60, 1200)`，在跳过结算动画的同时等待离开按钮出现。
4. **点击离开按钮**：`tap(1464, 792)`，等待 2 秒让页面切换。
5. **等待大厅特征并跳过升段**：使用 `retryTap(699, 852, isLobby, 60, 1000)`，持续点击升段跳过坐标直到**出现大厅特征**。
6. **二次确认**：出现大厅特征后等待 1500ms：
   - 若仍在大厅 → 成功。
   - 若不在大厅（加载慢导致先回大厅再进入升段页）→ 再次执行 `retryTap` 直到大厅特征重新出现。

`skipPromotion()` 用于处理已经处于升段奖励页的情况，流程与步骤 5、6 相同。

坐标说明：

- `leaveRetryTap = {1460, 812}`：等待离开按钮出现期间的点击位置。
- `leaveTap = {1464, 792}`：实际离开按钮点击位置。
- `promotionSkip = {699, 852}`：升段奖励跳过点击位置。

### 3.3 竞技场_路由.lua

处理从王国主城到竞技场大厅，以及离开竞技场回到主城的完整导航。

- `enter()`：主城 → 冒险页 → 点击「王国竞技场」→ 等待大厅
- `leave()`：关闭大厅 → 冒险页返回 → 等待回到主城

### 3.4 竞技场_任务.lua（状态机版）

任务编排层，使用 `core.state-machine` 实现竞技场流程。相比原始的过程式实现，状态机版将流程拆分为独立状态，每个状态只负责单一职责，便于定位问题、扩展功能与单步调试。

状态列表：

| 状态 | 职责 |
|---|---|
| `checkEnable` | 检查功能开关与 `maxBattles` 上限 |
| `enter` | 调用路由进入竞技场大厅 |
| `syncStatus` | 同步奖牌、门票、奖杯到会话 |
| `decideAction` | 决策中枢：扫荡、买票、升段重置、免费刷新、离开 |
| `autoBuy` | 自动购买门票 |
| `sweep` | 扫描当前页对手，过滤后进入战斗 |
| `battle` | 执行单场战斗 |
| `postBattle` | 战后统计、门票扣减、升段判定、继续扫荡 |
| `freeRefresh` | 等待并点击免费刷新 |
| `leave` | 同步最终状态并离开竞技场 |

主要行为：

1. 检查功能开关与战斗上限。
2. 进入竞技场并同步当前资源状态。
3. 门票不足时尝试自动买票。
4. 按页扫荡对手：
   - 第 1 页：位置 1、2、3
   - 第 2 页：位置 2、3
5. 跳过已战斗对手。
6. 按奖杯差阈值过滤对手。
7. 战斗后更新胜负平统计与门票数。
8. 检测到升段时右滑重置对手列表。
9. 两页扫完后触发免费刷新（若有）。
10. 安全离开竞技场。

### 3.5 竞技场_会话.lua

维护单次脚本运行期间的竞技场会话状态，使用 `lib.store` 持久化。

存储字段：

| 字段 | 说明 |
|---|---|
| `wins` | 胜利场次 |
| `losses` | 失败场次 |
| `draws` | 平局场次 |
| `tickets` | 当前门票数 |
| `buyCount` | 已自动买票次数 |
| `trophies` | 当前奖杯数 |
| `medals` | 奖牌数 |
| `nextFreeRefreshAt` | 下次免费刷新时间戳（秒） |

API：`get()` / `set(data)` / `update(partial)` / `totalBattles()` / `describe()` / `clear()` / `setNextFreeRefreshAt()` / `getTimeUntilRefresh()` / `clearNextFreeRefresh()`

## 4. 用户配置

配置入口位于 `帅斌饼干/脚本/config.lua` 的 `USER.arena`：

```lua
arena = {
    enabled = false,      -- 功能开关
    maxBattles = nil,     -- 战斗上限，nil 表示无限
    autoBuyCount = 0,     -- 自动买票次数，0 表示不买
    trophyDiff = 0,       -- 奖杯差值阈值
}
```

UI 配置位置：

- **功能开关**：`ui/tabs/feature-tab.lua` → 「王国竞技场」复选框
- **参数配置**：`ui/tabs/config-tab.lua` → 「王国竞技场」分组
  - 战斗上限（空 = 无限）
  - 自动买票次数
  - 奖杯差值阈值

## 5. 调度注册

在 `game/register.lua` 中注册为调度任务：

```lua
Scheduler.add("王国竞技场", conditionFn, taskFn)
```

调度条件：

1. `arena.enabled == true`
2. 矿山相关调度空闲（矿山优先）
3. 未达到 `maxBattles` 战斗上限

执行前若处于广场上下文，会先调用 `SquareTask.leaveForOtherTask()` 离开广场。

## 6. 核心流程说明

### 6.1 对手奖杯过滤逻辑

```lua
if myTrophy > info.trophies and not isClose(myTrophy, info.trophies, trophyDiff) then
    -- 跳过该对手
end
```

含义：仅当对手奖杯 **不显著低于** 我方时才开战。`trophyDiff` 用于控制「显著低于」的阈值。

### 6.2 升段判定

战斗结束后，先确保回到大厅，再读取**同一位置**的对手信息。注意：**升段判定必须在第2页左滑复位之前进行**，否则左滑可能改变当前视图，导致读取到错误的对手位而误判升段。

判定流程：

1. **未识别到对手位**（坐标为 0,0）：可能是列表正在刷新或识别失败，最多重试 3 次，超限后按升段处理。
2. **对手位存在但未标记「已战斗」**：可能是升段，也可能是颜色识别失败/列表自然刷新，同样最多重试 3 次，稳定显示未战后判定升段。
3. **对手位存在且已标记「已战斗」**：未升段，确认后才执行第2页左滑复位，并继续扫荡下一个位置。

判定为升段后，会右滑重置对手列表，并从第 1 页第 1 位重新开始扫荡。

#### 常见误判原因

| 原因 | 说明 |
|---|---|
| 第2页先左滑再判定 | 左滑后视图改变，原位置读到的是其他对手，自然未标记已战 |
| 结果颜色识别失败 | `resultOffset` 处颜色未匹配 `win/draw/lose`，可能是阈值太严、动画闪烁、偏移不准 |
| 对手列表自然刷新 | 游戏可能在没有升段的情况下刷新对手列表（如定时刷新） |
| 读取时机过早 | 回到大厅后列表尚未稳定 |

辅助调试信息：

- 战前/战后奖杯数变化会打印在日志中。
- 重试次数、是否已战、对手坐标均会输出。
- 日志关键字：`升段判定`、`对手位未识别`、`未标记已战`。

### 6.3 自动买票

当门票不足且 `autoBuyCount > 0` 且已买票次数未达上限时，调用 `buyTicket()` 购买门票。买票后再次同步状态，若门票仍大于 0 则继续扫荡。

### 6.4 免费刷新

两页对手扫荡完成后，**无论是否识别到「免费刷新」按钮**，都会进入 `freeRefresh` 状态：

1. **检测到「免费刷新」**：点击刷新按钮，清除 `nextFreeRefreshAt`，开启新一轮扫荡。
2. **未检测到「免费刷新」**：读取 `refreshOcr` 区域倒计时，支持解析：
   - `X分Y秒`
   - `X分`
   - `Y秒`
   - `MM:SS`
3. 将 `os.time() + seconds` 存入会话的 `nextFreeRefreshAt`。
4. 任务离开竞技场，等待下次调度。

> 注意：之前 `decideAction` 只在检测到「免费刷新」时才进入 `freeRefresh` 状态，否则直接离开，导致倒计时未被解析、刷新等待未生效。

解析失败时，默认 30 分钟后重试。

### 6.4.1 调度条件中的刷新等待

`game/register.lua` 中王国竞技场的 `condition` 会检查 `ArenaSession.getTimeUntilRefresh()`：

- 若倒计时未到期，在 HUD 显示 `刷新等待 Xs`，并跳过本轮。
- 若已到期或未设置，正常进入任务。

### 6.4.2 运行时空闲等待

`core/runtime.lua` 的无任务空闲等待已将竞技场刷新倒计时纳入计算：

```lua
waitRemain = math.max(farRemain, miningRemain, marketRemain, arenaRemain)
```

因此当竞技场刷新倒计时是最近的到期事件时，运行时会在该时间点前唤醒，优先执行竞技场任务。

### 6.5 状态机调试

状态机运行器 `core.state-machine` 会自动输出每轮 tick 的调试日志，包括当前状态、重试次数、异常次数等。

HUD 实时显示当前状态：

```
[sweep] 战斗3/∞ 胜2 负1 平0 胜率66.7% 门票2 买票0 奖杯1250
```

状态前缀 `[sweep]` 表示当前处于扫荡状态，便于在日志和悬浮窗中快速定位执行位置。

若要新增状态：

1. 在 `竞技场_任务.lua` 中编写新的状态处理函数。
2. 将其注册到 `handlers` 表中。
3. 从现有状态返回新状态名即可触发迁移。

## 7. 注意事项与维护提示

1. **UI 适配**：当游戏版本更新导致界面变化时，优先检查 `竞技场_特征库.lua` 中的颜色点和 OCR 区域。
2. **OCR 识别**：对手战力/奖杯使用专用 `numberOcr` 配置，识别失败通常与颜色、字号或背景变化有关。
3. **战斗超时**：结算页等待上限为 120 秒，若战斗动画或网络延迟较长可能需要调整。
4. **门票扣除**：门票在战斗完成后由任务模块手动减 1，实际购买次数由 `buyCount` 记录。
5. **会话清理**：用户可通过 UI 「重置」按钮清除 `arena_session`，重新开始统计。
